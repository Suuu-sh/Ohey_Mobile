# Ohey 説明資料

作成日: 2026-06-08  
対象: Ohey Mobile / Backend の現在のコードベース

## 1. 一言でいうと

**Oheyは、友だちと「今日空いてる？」「これやりたい」を、重くならずに共有できる予定・ゆるぼアプリです。**

予定をきっちり管理するカレンダーというより、

- 今日の空き具合をゆるく出す
- やりたいことを置いておく
- 友だちに気軽に声をかける
- 成立した予定やゆるぼを残す

という、日常の小さな誘いを生みやすくすることに寄せています。

## 2. 解決したい課題

### 課題

友だちを誘うとき、意外と心理的コストがあります。

- 「今日空いてる？」と聞くのが少し重い
- 断られると気まずい
- グループチャットで流れる
- 自分の予定感を毎回説明するのが面倒
- やりたいことはあるけど、誰を誘うか決まっていない

### Oheyの考え方

Oheyでは、予定を「確定イベント」だけでなく、
**空き具合・やりたいこと・ゆるい募集**として扱います。

これにより、誘う前の空気感を共有でき、友だちとの予定作りが軽くなります。

## 3. 主な機能

### 3.1 今日の予定ステータス

ユーザーは、その日の状態を次のように表せます。

- 空いてる
- たぶん空いてる
- 時間次第
- 空いてない
- まだ決めてない

このステータスは、フレンズ一覧やカレンダー、誘いの判断に使われます。
「予定の有無」を細かく入力しなくても、誘いやすさが伝わる設計です。

### 3.2 ゆるぼ

「誰か行かない？」「これやりたい」という投稿を作れる機能です。

ゆるぼには以下の情報を持たせられます。

- タイトル
- 本文
- カテゴリ
- 場所
- 時間ラベル
- 開始日時
- 公開範囲
- 参加リアクション

リアクションには、単なるいいねではなく、

- 興味あり
- 行ける
- 別日なら
- 承認待ち

のような予定調整に近い意味を持たせています。

また、ゆるぼは24時間で期限切れになる設計があり、一覧が古い募集で埋まらないようにしています。

### 3.3 やりたいことリスト

今すぐ誘うほどではないけれど、いつかやりたいことを保存できます。

- タイトル
- メモ
- カテゴリ
- 場所
- URL
- 公開範囲

を持てるため、友だちとの会話や誘いの種になります。

### 3.4 フレンズ / グループ

友だち申請、承認、お気に入り、グループ管理があります。

公開範囲は、

- 自分だけ
- 全フレンズ
- グループ

のように分けられます。

「全員に見せる」だけでなく、近い友だちだけに共有できるところが特徴です。

### 3.5 招待

特定の日付に対して友だちを誘えます。

招待には、

- 返信待ち
- 予定あり
- 見送り済み
- 取り消し済み

という状態があり、成立した予定をわかりやすく扱えます。

### 3.6 通知 / お知らせ

予定や投稿はとして残せます。

ゆるぼには、

- 日付
- 一緒にいたフレンズ
- 場所
- メモ
- リンク
- 位置情報
- いいね
- 通報 / 非表示

などがあります。

単なる予定管理で終わらず、「誰と何をしたか」をあとから振り返れる設計です。

### 3.7 通知 / Push

フレンド申請、招待、いいね、タグ付けなどに通知があります。

バックエンドには通知アウトボックスとPush配信の仕組みがあり、FCM経由でiOSにも通知できます。

### 3.8 Ohey Plus

Ohey Plusは、現在は広告非表示を中心にした有料プランです。

- RevenueCatで課金状態を管理
- App Storeサブスク `ohey_plus_monthly`
- Entitlement `plus`
- Plusユーザーは広告を表示しない

将来的に、Plus向け機能を拡張しやすい構成になっています。

### 3.9 広告

AdMobネイティブ広告に対応しています。

- Debug/ProfileではGoogle公式テスト広告ID
- Releaseでは本番AdMob ID
- 本番ビルドで誤ってテスト広告を出さないガード
- Plusユーザーには広告を出さない
- ATT / IDFA / App Privacyに対応

広告は投稿リストなどの体験を壊しにくい位置に差し込む設計です。

## 4. Oheyの強み

### 4.1 「予定管理」ではなく「誘いやすさ」に寄せている

多くのカレンダーアプリは、確定済みの予定を管理するためのものです。  
Oheyはその前段階、つまり「空いてるかも」「誰か行かない？」という曖昧な状態を扱います。

この曖昧さを許容することで、友だちを誘う心理的ハードルを下げています。

### 4.2 断りやすさも設計に入っている

「行ける」だけでなく、

- たぶん空いてる
- 時間次第
- 別日なら
- 見送り

のような中間表現があります。

誘う側にも誘われる側にも逃げ道があり、人間関係を壊しにくい設計です。

### 4.3 フレンド・グループ単位の公開範囲

予定やゆるぼは、誰に見せるかが重要です。  
Oheyでは自分だけ、全フレンズ、グループなどの公開範囲を持てるため、現実の人間関係に近い共有ができます。

### 4.4 見た目がポップで、予定共有の重さを減らしている

アバター、3Dボタン、カラフルなテーマ、マスコット背景などにより、予定調整の硬さをやわらげています。

機能としては予定・通知・権限管理を扱っていますが、UIは「ゆるく楽しい」印象に寄せています。

### 4.5 課金・広告・通知・審査対応まで実装済み

単なるプロトタイプではなく、以下の運用要素まで入っています。

- Supabase Auth
- Production / Dev環境分離
- Render Backend
- Push通知
- App Store Connect / TestFlight運用
- RevenueCatサブスク
- AdMob
- App Privacy / ATT対応
- 通報、ブロック、ミュート
- 管理者向け機能

公開・運用を見据えた構成になっています。

## 5. 工夫したところ

### 5.1 起動体験

起動時にOheyのオープニング画像を先読みし、最低表示時間とフェードアウトを設けています。

Supabase初期化、プロフィール読み込み、Plus状態確認、Push通知開始など、起動時に必要な処理を行いつつ、ユーザーには自然なオープニングとして見せます。

バックエンドがコールドスタートしても、完全に固まるのではなく、待機画面やリトライに流せるようにしています。

### 5.2 リアルタイム更新と再読み込みの制御

フレンド申請、招待、ゆるぼ参加などは、通知やRealtime更新の対象です。  
ただし、何度も更新するとUIが重くなるため、再読み込みのデバウンスや短時間の連続更新抑制を入れています。

### 5.3 Deep Link / 共有ゆるぼ

共有リンクからアプリを開き、対象のゆるぼに参加する導線があります。

対応している形式は、アプリスキームとWeb系URLの両方です。

- `app.ohey.com://...`
- `app.ohey.com.dev://...`
- `https://.../share/yurubos/...`

ログイン状態や画面状態を見ながら、参加処理と一覧遷移を行う設計です。

### 5.4 Plus判定をバックエンド状態とRevenueCat状態の両方で見る

Plus状態は、

- バックエンドの `isPlus`
- RevenueCatのEntitlement

のどちらかが有効ならPlusとして扱います。

これにより、課金反映の遅延や管理側付与にも対応しやすくなっています。

### 5.5 広告の安全な環境切り替え

広告設定では、Debug/ProfileとReleaseを明確に分けています。

- 開発中はテスト広告
- 本番は本番IDのみ
- 本番IDが空やサンプルIDなら広告無効
- 必要なときだけ `OHEY_ADMOB_FORCE_TEST_ADS` でTestFlight検証可能

本番審査で誤ってテスト広告を出す事故を防ぐ設計です。

### 5.6 Supabase RLSを活かしたBackend

Backendは、Supabase JWTを受け取り、そのユーザー権限のままSupabase/PostgRESTにアクセスします。  
これにより、RLSを活かしつつ、Backendで通知、管理、検証、複雑な処理を担当できます。

管理者機能だけはサービスロールを使うため、Flutterアプリには秘密鍵を入れず、信頼できるBackend経由に限定しています。

### 5.7 安全機能

ユーザー間アプリとして必要な安全機能を実装しています。

- ブロック
- ミュート
- 通報
- 投稿非表示
- 管理者画面
- 通報ステータス管理
- アカウント削除
- RLS

友だち向けアプリでも、公開後のトラブル対応を見据えています。

### 5.8 AI駆動開発しやすい設計方針

Backendには、Feature Slice型の軽量Clean Architecture方針があります。

目的は、AIや複数作業者が変更範囲を間違えにくくすることです。

- handlerを薄くする
- usecaseに操作単位を集める
- Supabase queryをrepositoryに寄せる
- domainに業務判断を置く
- テストを仕様として残す

今後の拡張でも、機能単位で安全に変更しやすい設計です。

## 7. 人に説明するときの短い話し方

### 30秒版

Oheyは、友だちと今日の空き具合や「これやりたい」をゆるく共有するアプリです。  
普通のカレンダーのように確定予定を管理するのではなく、誘う前の「空いてるかも」「誰か行かない？」を扱います。  
ステータス、ゆるぼ、やりたいことリスト、招待を通じて、友だちを誘う心理的ハードルを下げることを目指しています。

### 1分版

Oheyは、友だちとの予定作りを軽くするためのアプリです。  
ユーザーは今日の状態を「空いてる」「たぶん空いてる」「時間次第」などで出せて、友だちはそれを見ながら誘えます。  
また、「誰かカフェ行かない？」のようなゆるぼ投稿や、いつかやりたいことリストを作れるので、予定が確定する前の曖昧なコミュニケーションをアプリ上で扱えます。  
公開範囲はフレンズやグループ単位で調整でき、成立した予定や遊んだ記録はゆるぼとして残せます。  
課金、広告、Push通知、通報やブロックなど、公開後の運用も見据えて実装しています。

### エンジニア向け版

Flutter/RiverpodのモバイルアプリとGo Backend、Supabaseを組み合わせたソーシャル予定共有アプリです。  
Supabase AuthとRLSを軸に、BackendはJWTを使ってPostgRESTへアクセスしつつ、通知、Push、管理、複雑なAPI処理を担当します。  
Mobile側は日次ステータス、ゆるぼ、フレンズ、招待、Plus、広告を機能単位で構成し、RevenueCatとAdMobにも対応しています。  
Backendは今後のAI駆動開発を意識して、Feature Slice型の軽量Clean Architectureに寄せる方針を持っています。
